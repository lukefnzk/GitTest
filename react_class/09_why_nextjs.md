# 09. 브릿지 — "왜 Next.js 인가?"

`react_class` 의 **핵심 8개**로 React 의 심장(`상태를 바꾸면 화면이 알아서 따라온다`)을 익혔다.
마지막으로, **08번 레슨에서 우리가 직접 목격한 약점**을 짚고, Next.js 가 그걸 어떻게 없애는지 본다.
이걸 지금 이해하면 Next.js 학습이 "새 프레임워크 암기"가 아니라 "약점의 해결책"으로 들어온다.

> `web_class/10_why_react.md` 가 **"바닐라 JS의 불편함 → React"** 였다면,
> 이 문서는 **"순수 React의 약점 → Next.js"** 다. 같은 자리, 한 칸 위.

---

## 1. 직접 확인해보자 — 08번 레슨에서 `Ctrl+U`

`npm run dev` 로 08번 레슨을 띄운 뒤, 브라우저에서 **우클릭 → 페이지 소스 보기** (`Ctrl+U`).
개발자 도구(`F12`)의 Elements 탭이 아니다. **"소스 보기"** 여야 한다. 둘은 보여주는 게 다르다.

- **소스 보기** = 서버가 보내준 **원본 HTML** (크롤러가 보는 것)
- **Elements 탭** = JS 가 다 실행된 **현재 화면 상태** (사람이 보는 것)

화면엔 사용자 10명의 이름이 잘 나와 있는데, 소스에는 이것뿐이다.

```html
<div id="root"></div>
<script type="module" src="/src/main.jsx"></script>
```

**데이터가 하나도 없다.** 서버가 보낸 HTML 은 텅 비어 있었고,
그 뒤 브라우저가 JS 를 내려받아 실행하고 → `useEffect` → `fetch` →
`setUsers` → 자동 리렌더를 거쳐 화면을 채운 것이다.

- 사람은 잠깐 `⏳ 불러오는 중…` 을 보고 기다려 준다.
- **검색엔진 크롤러는 기다려 주지 않는다.** 빈 `<div>` 만 보고 떠난다.

이게 **CSR 의 약점**이고, 08번 레슨이 일부러 이렇게 만들어진 이유다.

---

## 2. SSR = 사실 PHP 로 평생 해온 것

**SSR = Server-Side Rendering.** "화면(HTML)을 **서버에서** 만들어 완성된 상태로 보낸다."

```php
<?php $posts = $db->query("SELECT * FROM posts"); ?>
<ul>
  <?php foreach ($posts as $p): ?>
    <li><?= $p['title'] ?></li>
  <?php endforeach; ?>
</ul>
```

이 코드는 **서버에서** 실행되고, 브라우저가 받는 건 이미 완성된 HTML 이다.

```html
<ul>
  <li>첫 번째 글</li>
  <li>두 번째 글</li>
</ul>
```

이게 SSR 이다. PHP 개발자는 **SSR 이라는 단어를 몰랐을 뿐, 처음부터 SSR 만 해왔다.**
SSR 이라는 이름이 따로 생긴 건, React 가 그 반대편(CSR)으로 갔다가 다시 돌아왔기 때문이다.

| | CSR (지금 `react_class`) | SSR (PHP, Next.js) |
|---|---|---|
| HTML 을 만드는 곳 | **브라우저** | **서버** |
| 브라우저가 받는 것 | 빈 껍데기 + JS 뭉치 | 내용이 다 채워진 HTML |
| 첫 화면 | 잠깐 하얗다가 나타남 | 즉시 보임 |
| 검색엔진이 보는 것 | 빈 `<div>` | 내용 전부 |
| 데이터 가져오기 | 브라우저가 API 호출 | 서버가 DB 직접 조회 |
| API 키 | 개발자 도구에 **노출** | 서버에만 있음 |

## 3. 하이드레이션 — Next.js 가 PHP 와 다른 지점

그럼 Next.js 는 그냥 PHP 로 돌아간 건가? 아니다. **둘의 장점을 합쳤다.**

```
1. 서버가 React 컴포넌트를 서버에서 실행 → 완성된 HTML 생성 → 전송
2. 브라우저: 즉시 내용을 그린다   ← 아직 버튼은 안 눌린다. 그냥 '그림'이다
3. JS 가 도착해 그 HTML 에 달라붙는다  → 버튼이 살아난다
4. 이후로는 SPA 처럼, 페이지를 옮겨도 새로고침 없이 부드럽게 동작
```

3번을 **하이드레이션(hydration, 수분 공급)** 이라 부른다.
**말라 있는 HTML 에 JS 라는 물을 부어 살려낸다**는 비유다.

> PHP 는 버튼을 누를 때마다 페이지가 통째로 새로고침됐다.
> Next.js 는 **첫 화면만 PHP 처럼** 주고, 그 뒤로는 **React 로 살아간다.**

## 4. 렌더링 전략 4가지 — 약자 풀어보기

Next.js 의 진짜 힘은 SSR 하나가 아니라, **페이지마다 아래 넷 중 하나를 골라 쓸 수 있다**는 점이다.

| 약자 | 영문 전체 | 한국어 | 글자 그대로의 뜻 |
|---|---|---|---|
| **CSR** | **C**lient-**S**ide **R**endering | 클라이언트 사이드 렌더링 | **브라우저 쪽**에서 화면을 만든다 |
| **SSR** | **S**erver-**S**ide **R**endering | 서버 사이드 렌더링 | **서버 쪽**에서 화면을 만든다 |
| **SSG** | **S**tatic **S**ite **G**eneration | 정적 사이트 생성 | **미리 만들어 둔** 고정된 페이지 |
| **ISR** | **I**ncremental **S**tatic **R**egeneration | 증분 정적 재생성 | 미리 만들어 두되 **조금씩 다시 만든다** |

### 단어를 하나씩 뜯으면

- **Rendering(렌더링)** — 데이터를 가지고 **최종 화면(HTML)을 만들어 내는 행위**.
  PHP 에서 `foreach` 로 `<li>` 를 찍어내던 그 작업이 바로 렌더링이다.
- **Client(클라이언트)** — 서버에 요청을 보내는 쪽, 즉 **브라우저**.
- **Static(정적)** — "고정되어 안 변한다". 반대말은 Dynamic(동적).
  `.php` 는 요청마다 결과가 다르니 동적, 그냥 올려둔 `.html` 은 누가 봐도 같으니 정적.
- **Generation(생성)** vs **Regeneration(재생성)** — SSG 는 **한 번 만들고 끝**,
  ISR 은 만들어 둔 걸 **나중에 다시** 만든다. 그래서 `Re-` 가 붙는다.
- **Incremental(증분)** — "전부 한꺼번에가 아니라, 필요한 것만 조금씩".
  상품 10만 개짜리 쇼핑몰에서 **가격이 바뀐 한 페이지만** 다시 굽는다.

### 두 축으로 나뉜다

- **CSR / SSR** 은 **"어디서 만드나"** 를 묻는다 → Client 냐 Server 냐. (뒤는 똑같이 Rendering)
- **SSG / ISR** 은 **"언제 만드나"** 를 묻는다 → 둘 다 서버에서 만들지만,
  요청이 올 때가 아니라 **미리(빌드할 때)** 만들어 둔다. 차이는 그 뒤에 갱신하느냐(ISR) 마느냐(SSG).

## 5. 빵집 비유

| | 비유 |
|---|---|
| **SSR** | 손님이 올 때마다 **그 자리에서 굽는다** (항상 신선, 조금 기다림) |
| **SSG** | **새벽에 다 구워 진열해 둔다** (즉시 집어감, 종일 그 빵) |
| **ISR** | 진열해 두되 **한 시간마다 새로 굽는다** (거의 즉시 + 적당히 신선) |
| **CSR** | 손님에게 **밀가루와 오븐을 준다** (본인이 굽는 동안 기다림) |

마지막 CSR 이 왜 SEO 에 불리한지가 이 비유로 선명해진다.
**크롤러는 밀가루만 받아 들고 "여긴 빵이 없네" 하고 가버린다.**

어디에 쓰나:

| 전략 | 어울리는 곳 |
|---|---|
| CSR | 로그인 후 대시보드 (검색 노출이 필요 없다) |
| SSR | 실시간으로 바뀌는 상품 상세, 로그인 상태별 화면 |
| SSG | 블로그 글, 회사 소개, 문서 |
| ISR | 뉴스 목록, 자주는 아니어도 가끔 바뀌는 목록 |

## 6. 핵심 한 줄

> Next.js = **"페이지마다 언제·어디서 HTML 을 만들지 고를 수 있게 해주는"** React 프레임워크.

순수 React 는 선택지가 CSR 하나뿐이었다. 그래서 08번 레슨에서 소스가 비어 있었던 것이다.

## 7. 다음에 배울 것 (Phase 2 — Next.js)

1. **라우팅** — 폴더 구조가 곧 URL 이 된다. `react_class` 의 `App.jsx` "리모컨"이 사라진다.
   (PHP 에서 `/about.php` 파일이 곧 `/about` 주소였던 것과 닮았다)
2. **렌더링 전략 선택** — 위 4가지를 페이지마다 지정한다.
3. **서버에서 데이터 가져오기** — `useEffect` + `fetch` 대신 **서버가 직접 DB 를 읽는다.**
   API 키가 브라우저에 노출되지 않는다.
4. **SEO** — `<title>`, `<meta>` 를 페이지마다 서버에서 심는다. 크롤러가 내용을 읽는다.
5. **배포** — 빌드 결과가 정적 파일이냐, Node 서버가 필요하냐가 여기서 갈린다.

> 배포 이야기(Vercel 이 필수인가, 직접 호스팅은 어떻게)는 **Node 서버가 필요해지는 시점**과
> 엮여 있어서 Phase 2 에서 다룬다. 지금 `react_class` 의 `npm run build` 결과(`dist/`)는
> **순수 정적 파일**이라 아무 웹서버(지금 쓰는 Apache/Nginx 포함)에나 올라간다는 것만 기억하자.
